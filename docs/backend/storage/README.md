# PrintForge - Designer S3 Flow

## Overview

- Design state lives entirely on the client (browser) during the design session
- Files are uploaded to S3 when the customer clicks Add to Cart - not during design, not at checkout
- Each cart item has its own unique `sessionId` and its own folder in S3
- The flow is fully reliable after upload - no browser dependency once files are in S3

---

## S3 Structure

```
temp/
  {sessionId}/                ← uploaded at Add to Cart, before order exists
    view-{viewId}.png
orders/
  {orderId}/
    {sessionId}/              ← moved here after order is confirmed
      view-{viewId}.png
```

---

## Full Flow

### 1. Customer designs in the designer iframe

- All design state (canvas elements, text, uploaded images) is kept in the browser only - in `localStorage` under key `printforge:designer:session:{sessionId}`
- Nothing is sent to the server or S3 during this phase
- The `sessionId` is a UUID generated by WordPress at page render time and passed to the iframe as a URL parameter (`?sessionId=...`)

### 2. Customer clicks Add to Cart

- The `printforge-configurator` WordPress plugin intercepts the cart form submit
- The plugin requests a PNG preview from the designer iframe via `postMessage`:
  - Sends: `printforge:designer:preview-request` with a `requestId`
  - Designer responds: `printforge:designer:preview-response` with PNG `data:image/png;base64,...` for each view
- The plugin converts each PNG data URL to a blob and uploads it directly to the Fastify backend:

```
POST /api/storage/temp/{sessionId}
Content-Type: multipart/form-data

file: <PNG blob>
viewId: <view identifier>
```

- Backend stores each file in S3 at: `temp/{sessionId}/view-{viewId}.png`
- If the upload fails, the form is submitted anyway - the customer is not blocked from purchasing (the design file is simply missing)
- The plugin sets a hidden form field `printforge_session_id` with the session UUID and submits the cart form
- WooCommerce adds the article to the cart with `_printforge_session_id` stored in the item meta

### 3. Customer checks out

- WooCommerce creates the order and assigns it an `orderId`
- The `printforge_configurator_assign_designs_on_checkout` WordPress hook fires
- The plugin calls the Fastify backend to move the files from temp to the permanent order folder:

```
POST /api/storage/orders/{orderId}/assign
x-printforge-secret: <webhook secret>
Content-Type: application/json

{ "sessionIds": ["<uuid>", ...] }
```

### 4. Fastify moves the files

For each `sessionId` in the request body, Fastify:

1. Lists all objects under `temp/{sessionId}/` with `ListObjectsV2Command`
2. Copies each object to `orders/{orderId}/{sessionId}/` with `CopyObjectCommand`
3. Deletes the originals from `temp/` with `DeleteObjectCommand`
4. Returns `{ movedCount, keys[] }`

## Downloading Designs from the WooCommerce Admin

The `printforge-configurator` plugin renders design download links per order line item. It calls two Fastify endpoints, both protected with the webhook secret:

```
GET /api/storage/orders/{orderId}/{sessionId}
x-printforge-secret: <webhook secret>
→ Returns: [{ filename, size }, ...]

GET /api/storage/orders/{orderId}/{sessionId}/{filename}
x-printforge-secret: <webhook secret>
→ Streams the file back
```

---

## Authentication

Server-to-server calls from the WordPress plugin to the Fastify backend are authenticated with a shared secret sent in the `x-printforge-secret` request header. See the [Authorization doc](../authorization/README.md) for details.

The `POST /api/storage/temp/{sessionId}` upload endpoint is public - it is called directly from the customer's browser and requires no authentication.

## Why Upload at Add to Cart - Not Later

### Why not keep designs in localStorage until purchase?

The browser is not reliable enough for something the customer has already paid for. Failure scenarios:

- Customer pays, lands on the thank-you page, and closes the tab before any upload runs
- Browser crashes after payment completes
- Mobile Safari clears storage mid-session
- Customer switches devices between designing and paying

In all of these cases the order exists, the payment went through, and the design is gone. For a print shop, the design is the actual product the customer paid for - losing it is not acceptable.

### Why Add to Cart is the right moment

- The customer is not in a hurry
- No payment has happened yet - nothing is at stake
- It is a clean, unambiguous trigger with no competing processes
- If the upload fails, the customer can be informed immediately before they proceed
- The design is safely in S3 long before any money changes hands
